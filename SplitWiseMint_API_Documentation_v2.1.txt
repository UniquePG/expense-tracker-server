What Changed in v2.1.0 (Audit Corrections)

   This version corrects gaps identified during audit. Key changes are listed below.



Critical Additions
  •   POST /transactions/expense — NEW endpoint for simple personal expense tracking (the biggest missing
      gap). Allows recording spend with category, date, description directly against an account balance without
      requiring splits.
  •   PUT /transactions/:id — NEW endpoint to edit income or expense transaction details (description, date,
      category, amount).
  •   expenseType field added to POST /expenses — must send PERSONAL or GROUP. Auto-derived if
      omitted (groupId present = GROUP, else PERSONAL).
  •   Personal (solo) expense flow clarified — POST /expenses with just the current user in participants[]
      creates a self-only expense record (useful for categorised split history). Not required for simple tracking;
      use POST /transactions/expense instead.


Logic Fixes
  •   AccountTransaction categoryId — documented correctly; expense-type transactions can carry categoryId
      for category-level analytics.
  •   POST /expenses payer logic — clarified that paidById can be omitted and defaults to the authenticated
      user.
  •   Budget check trigger — clarified it runs against expense_splits for split expenses AND against
      account_transactions for personal transactions.
  •   DELETE /accounts/:id — corrected safety check wording; block if unsettled expense_splits reference this
      account via the linked expense's accountId column.


Minor Clarifications
  •   Section 3 (Accounts): GET /accounts/:id response includes category info on expense-type transactions.
  •   Section 6 (Expenses): GET /expenses includes both personal and split expenses where user is participant
      or payer.
  •   Section 15 (Analytics): income-vs-expense endpoint now correctly documents that personal expenses from
      account_transactions are included in the expense sum.

1.1 POST /auth/register
[POST] /auth/register                 Register a new user account
Used on screen: Registration Screen
Purpose: Create a new user account, hash password, issue JWT tokens, send email verification.


Request Body
 Field                      Type                 Required       Validation Rules
 email                      string               Required       Valid email format. Must be unique across users table.
 password                   string               Required       Min 8 chars. Must contain: 1 uppercase, 1 lowercase, 1
                                                                number.
 firstName                  string               Required       Min 2 chars, max 50 chars. No special characters.
 lastName                   string               Required       Min 2 chars, max 50 chars.
 phone                      string               Optional       E.164 format (+91XXXXXXXXXX). Must be unique if
                                                                provided.
 currency                   string               Optional       ISO 4217 code (INR, USD, EUR, etc.). Defaults to INR.



Business Logic (step by step)
    1.    Check uniqueness: SELECT from users WHERE email = ? OR phone = ?. Return 409 if exists.
    2.    Hash password using bcrypt (salt rounds = 12).
    3.    INSERT into users table. Store hashed password, not plain text.
    4.    Generate: accessToken (JWT, expires 15 min), refreshToken (UUID v4, expires 30 days).
    5.    INSERT into refresh_tokens (token, userId, expiresAt = now + 30 days).
    6.    INSERT into email_verifications (userId, token = secure random hex, expiresAt = now + 24h).
    7.    Send verification email with link: https://app.splitwisemint.com/verify?token=X
    8.    System categories are global (isSystem=true, userId=null) — no INSERT needed per user, fetched
          globally.

Tables Used: users → refresh_tokens → email_verifications


Response — 201 Created
    { "success": true, "message": "Account created. Please verify your email.",
      "data": { "user": { "id": "uuid", "email": "john@gmail.com", "firstName": "John",
        "lastName": "Doe", "currency": "INR", "isEmailVerified": false },
      "accessToken": "eyJ...", "refreshToken": "550e8400-..." } }

    transaction record.

                                                               Wallet'). Max 100 chars. Unique per user.

 balance                    number              Required       Initial balance. Can be negative (credit card).

4. Income & Transaction APIs
    These APIs manage the personal financial ledger. All income/expense/transfer entries update the linked account
    balance atomically. This is the core personal finance management section — the most important section for simple
    day-to-day tracking.


The two primary personal finance actions are:
   • POST /transactions/income — Record money coming IN to an account.
   • POST /transactions/expense — Record money going OUT from an account. (NEW — critical MVP feature)



4.1 POST /transactions/income
[POST] /transactions/income                       Record income into an account
Used on screen: Add Income Screen
 Field                         Type                 Required       Validation Rules
 accountId                     string (uuid)        Required       Target account to credit income into.
 amount                        number               Required       Income amount. Must be > 0. Decimal(12,2).
 categoryId                    string (uuid)        Optional       Category tag for this income (e.g., a custom "Salary"
                                                                   category). Must belong to system or current user.
 incomeSource                  string               Optional       Label: Salary | Freelance | Business | Rental | Gift |
                                                                   Interest | Dividend | Refund | Other. Free text also
                                                                   accepted.
 description                   string               Optional       Optional custom description. Max 255 chars.
 transactionDate               datetime             Optional       When income was received. Defaults to now(). Must
                                                                   not be a future date.

Income Source Suggestions (quick-select chips in UI)
Salary | Freelance/Contract | Business Income | Rental Income | Investment Return | Gift | Interest | Tax Refund |

4.2 POST /transactions/expense
[POST] /transactions/expense                       Record a personal expense from an account ✦ NEW
    ✦ This is a NEW endpoint added in v2.1.0. It was missing from v2.0.0 and is the primary way a user records simple
    personal spending (like groceries, fuel, a coffee) directly against an account balance without any bill-splitting.


Used on screen: Add Expense Screen (personal finance view), Quick Add Expense button
Purpose: The simplest way to record "I spent ₹X on Y today." Deducts from account, categorises, and feeds
budget tracking. No splits involved.


 Field                         Type                 Required       Validation Rules
 accountId                     string (uuid)        Required       Account to deduct expense from. Must belong to
                                                                   current user.
 amount                        number               Required       Expense amount. Must be > 0. Decimal(12,2).
 categoryId                    string (uuid)        Optional       Category for this expense. Must belong to system or
                                                                   current user. Strongly recommended for budget
                                                                   tracking.
 description                   string               Optional       What was spent on. Max 255 chars. e.g., "Lunch at
                                                                   Haldirams".
 transactionDate               datetime             Optional       When the expense occurred. Defaults to now().
                                                                   Cannot be more than 1 year in the past.



Business Logic (Atomic — DB Transaction)
    80. Verify accountId belongs to current user. Return 403 if not.
    81. Validate amount > 0.
    82. If categoryId provided: verify it belongs to system (isSystem=true) or current user.
    83. BEGIN TRANSACTION.
    84. INSERT into account_transactions: { userId, accountId, type=EXPENSE, amount, categoryId, description,
        transactionDate }.
    85. UPDATE accounts SET balance = balance - amount, updatedAt = now() WHERE id = accountId.
    86. COMMIT. Return the created transaction.
    87. ROLLBACK on any error.
    88. After commit (async): run budget check for this user (see Section 12.6). If categoryId matches an active
        budget and threshold crossed, fire BUDGET_ALERT notification.

Tables Used: account_transactions → accounts → budgets → notifications


Response — 201 Created
    { "success": true, "data": { "transaction": { "id": "uuid", "type": "EXPENSE",
      "amount": "450.00", "description": "Lunch at Haldirams",
      "transactionDate": "2026-03-22T13:00:00Z",
         "category": { "id": "uuid", "name": "Food & Dining", "icon": "🍔" },
         "account": { "id": "uuid", "name": "HDFC Savings", "balance": "49550.00" } } } }

4.3 POST /transactions/transfer
[POST] /transactions/transfer                        Transfer money between own accounts
Used on screen: Transfer Screen
 Field                           Type               Required       Validation Rules
 fromAccountId                   string (uuid)      Required       Source account. Must belong to current user.
 toAccountId                     string (uuid)      Required       Destination account. Must belong to current user.
                                                                   Must differ from fromAccountId.
 amount                          number             Required       Transfer amount > 0.
 description                     string             Optional       Transfer note. Max 255 chars.
 transactionDate                 datetime           Optional       Defaults to now().

Business Logic (Atomic)
   89. Verify both accounts belong to current user.
   90. Validate fromAccountId ≠ toAccountId.
   91. BEGIN TRANSACTION:
       ◦ INSERT account_transactions: type=EXPENSE, accountId=fromAccount, description='Transfer out:
          '+description, toAccountId=toAccountId.
       ◦ INSERT account_transactions: type=INCOME, accountId=toAccount, description='Transfer in from:
          '+fromAccount.name, toAccountId=fromAccountId.
       ◦ UPDATE accounts SET balance = balance - amount WHERE id = fromAccountId.
       ◦ UPDATE accounts SET balance = balance + amount WHERE id = toAccountId.
   92. COMMIT.
Tables Used: account_transactions → accounts

6. Expense APIs
    Expenses are the split-billing core feature. Use POST /transactions/expense (Section 4.2) for simple personal
    spending. Use POST /expenses when you need to split an amount with friends/contacts or track a group expense.


Creating an expense ALWAYS creates expense_split rows for each participant. The payer's own split is auto-
marked as settled (isPayer=true, isSettled=true). Account balance updates are atomic.


SPLIT CALCULATION RULES (Applied Server-Side)
   •     EQUAL: splitAmount = ROUND(totalAmount / count, 2). Remainder goes to payer's split.
   •     PERCENTAGE: splitAmount = ROUND(totalAmount × percentage / 100, 2). Sum must equal 100
         (validated).
   •     EXACT: Use provided amounts directly. Validate SUM = totalAmount exactly.
   •     SHARES: totalShares = SUM(all). Each amount = ROUND(totalAmount × shares / totalShares, 2).
         Remainder to payer.
   •     Payer's split: Always isPayer=true and isSettled=true (they already paid).
   •     Others' splits: isSettled=false, settledAt=null until settled.

7. Split & Balance APIs
Read-only calculation APIs. Compute balances from the expense_splits table in real time. No writing — pure

7.1 GET /splits/balances
[GET] /splits/balances                Get balance summary with all friends
Used on screen: Dashboard balance card, Friends Screen balance list

Business Logic — Correct Query
For each friend, calculate:
   • theyOweMe: SUM(es.amount) FROM expense_splits es JOIN expenses e WHERE es.userId = friend.id
      AND e.paidById = me AND es.isSettled = false
   • iOweThem: SUM(es.amount) FROM expense_splits es JOIN expenses e WHERE es.userId = me AND
      e.paidById = friend.id AND es.isSettled = false
   • netBalance: theyOweMe - iOweThem. Positive = they owe me. Negative = I owe them.
Tables Used: expense_splits → expenses → friendships → users



7.2 GET /splits/balances/simplified
[GET] /splits/balances/simplified                        Get minimum transactions to settle all debts

Debt Simplification Algorithm
   121.        Build net balance array: [{userId, netBalance}] where positive = owed to them, negative = they
       owe others.
   122.        Separate into creditors (positive net) and debtors (negative net).
   123.        While creditors and debtors both have entries: take largest creditor (C) and largest debtor (D).
       settleAmount = MIN(|C.balance|, |D.balance|). Output: "D pays C: settleAmount". Reduce both balances.
       Remove if zero.
   124.        Return the minimum list of payment instructions.
Tables Used: expense_splits → expenses → users

[GET] /budgets             Get all active budgets with current progress

Business Logic — Progress Calculation
For each budget, calculate current period spend from BOTH sources:
   • Source 1 — Split expenses: SELECT SUM(es.amount) FROM expense_splits es JOIN expenses e
      WHERE es.userId = me AND e.expenseDate IN period AND es.isSettled = false AND (categoryId matches

      me AND type = EXPENSE AND transactionDate IN period AND (categoryId matches if set).
   • currentSpend = Source1 + Source2. percentUsed = (currentSpend / budget.amount) × 100. remaining =
      budget.amount - currentSpend.
